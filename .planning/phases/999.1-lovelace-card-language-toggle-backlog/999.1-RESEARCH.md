## User Constraints (from CONTEXT.md)

### Locked Decisions
### Language Selector UX
- **D-01:** Place the language switcher in the card header as an explicit `SK/EN` segmented control.
- **D-02:** Switching is one-tap direct toggle between Slovak and English.
- **D-03:** Selector is placed inside a collapsible settings panel instead of always-on display.

### Persistence Policy
- **D-04:** Persist language choice in card frontend storage using a dedicated `localStorage` key.
- **D-05:** On first load (no saved preference), detect from `hass.language` once and immediately persist the inferred choice.
- **D-06:** If stored value is missing/invalid, re-run first-load detection and overwrite with a valid value.

### Translation Architecture
- **D-07:** Use one typed runtime dictionary object keyed by language and translation key.
- **D-08:** Enforce strict SK/EN key parity via shared TypeScript contracts.
- **D-09:** Missing runtime key falls back to English for that key while continuing to render.

## Summary

The current card already uses Lit reactive rendering and has proven `localStorage` persistence (`flynow.last_balloon`), so language toggling should be implemented as a small extension of existing card state patterns rather than a new architecture. [VERIFIED: codebase `lovelace/flynow-card/src/flynow-card.ts`] Lit's reactive property model explicitly supports update-triggering state changes, which fits instant language switching in one render cycle. [CITED: https://lit.dev/docs/components/properties/]

`localStorage` is the right persistence layer for this scope because settings persist across browser sessions for the same origin, matching the requirement to honor user-selected language after first load. [CITED: https://developer.mozilla.org/en-US/docs/Web/API/Window/localStorage] The implementation should use first-load detection from `hass.language` only when storage is absent/invalid, then persist and always read from storage first. [VERIFIED: `999.1-CONTEXT.md` decisions D-04..D-06]

**Primary recommendation:** Add a reactive `selectedLanguage: "sk" | "en"` state in `flynow-card.ts`, initialize it via `localStorage -> hass.language fallback`, then route all UI strings through a typed `TRANSLATIONS` dictionary with English per-key fallback. [VERIFIED: codebase + context decisions]

## Architecture Patterns

### System Architecture Diagram

```text
HA user profile locale (hass.language)
            |
            v
Card init -> Read localStorage key ----invalid/missing----> infer from hass.language -> persist
   |                    |                                            |
   |                    +----------------valid------------------------+
   v
selectedLanguage reactive state ("sk" | "en")
            |
            v
t(key) helper -> typed dictionary lookup -> key missing? -> English fallback for key
            |
            v
All card sections render translated labels/messages
            |
            v
User taps SK/EN toggle -> update state + persist -> immediate rerender
```

### Recommended Project Structure
```text
lovelace/flynow-card/src/
├── flynow-card.ts      # reactive language state + UI wiring
├── i18n.ts             # TRANSLATIONS dictionary + helper types/functions
└── types.ts            # shared unions/types (LanguageCode, TranslationKey)
```

### Pattern 1: Reactive Language State with Initialization Guard
**What:** Keep a single reactive language state in card instance and initialize once from storage/fallback.  
**When to use:** Any persisted UI preference in this card (language, future display toggles).  
**Example:**
```typescript
// Source: https://lit.dev/docs/components/properties/
static properties = {
  hass: { attribute: false },
  selectedLanguage: { state: true },
};
```

### Pattern 2: Typed Dictionary with Key Parity
**What:** Define `TranslationKey` and `Record<LanguageCode, Record<TranslationKey, string>>`.  
**When to use:** Small fixed-language card text where compile-time parity matters.  
**Example:**
```typescript
type LanguageCode = "sk" | "en";
type TranslationKey = "section" | "surfaceWind" | "visibility";
const TRANSLATIONS: Record<LanguageCode, Record<TranslationKey, string>> = { /* ... */ };
```

### Pattern 3: Safe Fallback Translator
**What:** `t(key)` returns selected-language value or English key fallback.  
**When to use:** When runtime resilience is required (D-09).  
**Example:**
```typescript
const fallback = TRANSLATIONS.en[key];
return TRANSLATIONS[this.selectedLanguage]?.[key] ?? fallback;
```

### Anti-Patterns to Avoid
- **Mixed text sources:** Leaving hardcoded English/Slovak literals in templates causes partial translation drift; route all user-facing text through dictionary helpers. [VERIFIED: current literals in `flynow-card.ts`]
- **Deriving language every render from `hass.language`:** This breaks user override persistence and violates D-05/D-06 behavior. [VERIFIED: context decisions]
- **Untyped dictionary keys:** Using untyped string maps allows silent key mismatch between SK/EN. [ASSUMED]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Reactive rerendering | Manual DOM patching | Lit reactive properties/state | Lit already handles update scheduling and render lifecycle. [CITED: lit properties docs] |
| Preference persistence | Custom storage abstraction | Native `window.localStorage` | Built-in API is sufficient for per-origin persisted preference. [CITED: MDN localStorage] |
| Translation key validation | Runtime key scanners only | TypeScript union-based key contracts | Compile-time parity catches drift earlier. [ASSUMED] |

**Key insight:** This phase is a UI state/presentation change; reuse existing Lit + TypeScript + localStorage patterns already present in the card instead of introducing external localization systems. [VERIFIED: codebase]

## Common Pitfalls

### Pitfall 1: Incomplete String Coverage
**What goes wrong:** Some labels/messages remain untranslated after toggle.  
**Why it happens:** Text literals are scattered in templates and helper methods.  
**How to avoid:** Create a translation key inventory first, then replace all visible strings through `t(key)`.  
**Warning signs:** Mixed-language UI after switching.

### Pitfall 2: Preference Not Persisted Deterministically
**What goes wrong:** Language flips back after refresh.  
**Why it happens:** Initialization order is wrong or storage key invalid handling is missing.  
**How to avoid:** Use deterministic init order: `read -> validate -> fallback detect -> persist`.  
**Warning signs:** Refresh causes unexpected language changes.

### Pitfall 3: Non-reactive State Updates
**What goes wrong:** Toggle updates storage but UI does not refresh immediately.  
**Why it happens:** State kept in non-reactive field without `requestUpdate`.  
**How to avoid:** Use Lit internal reactive state (`state: true`) or always call `requestUpdate()` after mutation. [CITED: lit properties docs]
**Warning signs:** UI updates only after unrelated state changes.
